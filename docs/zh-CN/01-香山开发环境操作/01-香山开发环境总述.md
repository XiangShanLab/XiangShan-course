style="color:#000000;">功能正确</font>\*\*<font style="color:#000000;">到</font>\*\*<font style="color:#000000;">性能探索</font>\*\*<font style="color:#000000;">，最终落地于</font>\*\*<font style="color:#000000;">高质量实现</font>\*\*<font style="color:#000000;">的技术路径。这三者层层递进，共同构成了香山作为开源高性能处理器项目的核心开发理念。</font>  ### <font style="color:#000000;">1. 指令级功能实现的正确性：大厦的地基</font> <font style="color:#000000;">这是处理器开发的起点，也是所有工作的前提。</font>  + \*\*<font style="color:#000000;">功能正确</font>\*\*<font style="color:#000000;">：无论微架构多么先进，首先必须保证处理器能够准确无误地执行指令集中的每一条指令。</font> + \*\*<font style="color:#000000;">验证驱动</font>\*\*<font style="color:#000000;">：在香山的开发中，这一层面通常通过大量的指令测试集（如RISC-V 架构测试套件）和形式化验证来保证。只有确保了指令级的功能正确，后续的性能优化才有意义。</font>  ### <font style="color:#000000;">2. 微架构性能探索：创新的引擎</font> <font style="color:#000000;">在功能正确的基础上，香山作为一个科研与工程结合的产物，非常注重微架构层面的创新与探索。</font>  + \*\*<font style="color:#000000;">寻找性能瓶颈</font>\*\*<font style="color:#000000;">：探索意味着分析现有设计的不足。例如，前端取指带宽是否足够？分支预测器是否足够精准？数据预取能否更智能？</font> + \*\*<font style="color:#000000;">架构空间探索</font>\*\*<font style="color:#000000;">：通过模拟器和性能模型，尝试不同的缓存层次结构、流水线深度、乱序执行窗口大小等参数，寻找性能、功耗与面积的帕累托最优解。</font> + \*\*<font style="color:#000000;">南湖/昆明湖架构</font>\*\*<font style="color:#000000;">：香山的历代架构（如南湖、昆明湖）本身就是这种探索的成果，它们代表了针对不同目标场景（如桌面级、服务器级）的微架构设计尝试。</font>  ### <font style="color:#000000;">3. 高性能RTL的实现：从图纸到现实</font> <font style="color:#000000;">这是将微架构的构想转化为实际可流片、可运行的硬件电路的过程。</font>  + \*\*<font style="color:#000000;">可综合的代码</font>\*\*<font style="color:#000000;">：RTL（寄存器传输级）代码不仅仅是软件模拟，它必须能够被逻辑综合工具转换成实际的门级网表。这要求代码风格严谨，时序清晰。</font> + \*\*<font style="color:#000000;">敏捷开发语言</font>\*\*<font style="color:#000000;">：香山使用 Chisel（一种嵌入在Scala中的硬件构造语言）来生成RTL。这种高级语言允许开发者以更高效、更参数化的方式描述复杂的硬件逻辑，从而支撑起高性能设计的复杂度。</font> + \*\*<font style="color:#000000;">PPA（性能、功耗、面积）调优</font>\*\*<font style="color:#000000;">：在RTL层面，需要精细地控制关键路径的延迟、优化功耗，并合理布局，确保最终流片出来的芯片能够在目标频率下稳定运行。</font>  ---  \*\*<font style="color:#000000;">总结来看，香山处理器的开发逻辑是：</font>\*\*  1. \*\*<font style="color:#000000;">首先要“能用”</font>\*\*<font style="color:#000000;">（指令正确）；</font> 2. \*\*<font style="color:#000000;">其次要“好用”</font>\*\*<font style="color:#000000;">（微架构探索，追求更高性能）；</font> 3. \*\*<font style="color:#000000;">最终要“能做出来”</font>\*\*<font style="color:#000000;">（高性能RTL实现，确保可流片）。</font>  <font style="color:#000000;">这种三位一体的模式，使得香山不仅是一个学术界的研究平台，也是一个具备工业级潜力的高性能处理器核。</font>  ## 1.3 Workload 运行流程 ### 1.4.1 什么是 Workload？ Workload 是CPU运行的程序，比如简单的 "Hello XiangShan" 或复杂的性能测试程序。在香山环境中，有两种主要方式运行 Workload：  ### 方式一： ### 使用 NEMU 运行（快速测试） \*\*适用场景：\*\* 快速验证程序逻辑，不涉及硬件细节   \*\*流程：\*\*  1. 使用 nexus-am 生成 Workload 程序 2. 编译 NEMU 模拟器 3. 在 NEMU 中运行程序  \*\*优点：\*\* 速度快，几秒内完成   \*\*缺点：\*\* 不验证硬件实现细节  ### 方式二： ### 使用香山 RTL 仿真运行（真实测试） \*\*适用场景：\*\* 验证硬件设计正确性   \*\*流程：\*\*  1. 生成 Workload 程序 2. 将 Chisel 设计转换为 Verilog 代码 3. 使用 Verilator 编译成可仿真程序 4. 运行仿真，验证结果  :::danger \*\*特别推荐：使用 DiffTest 模式\*\*  DiffTest 是\*\*香山环境的一大特色\*\*，它让 NEMU（软件模拟器）和香山 RTL（硬件设计）同时运行同一个程序，并实时比对两者的状态。这就像有一个经验丰富的老师在旁边，每一步都检查你的答案是否正确。  :::    \*\*DiffTest 工作流程：\*\*  \`\`\`plain 程序 → NEMU（参考模型） → 状态比对 → 报告差异          ↓ 程序 → 香山 RTL（待验证） → 状态比对 → 报告差异 \`\`\`  如果两者状态始终一致，说明硬件设计正确；如果出现差异，就需要调试硬件设计。  ## 1.5 新手实战 - 运行第一个 Hello XiangShan 这是你的第一个实战任务！跟着下面的步骤，你将成功在香山环境中运行第一个程序。  ### 1.5.1 第一步：环境准备 \`\`\`bash # 1. 克隆环境仓库 git clone + io.src(1)/' src/main/scala/xiangshan/backend/fu/Alu.scala  # 重新编译仿真程序 (这一步很重要 在每次更新 RTL 代码后都需要重新执行这条指令) make emu CONFIG=MinimalConfig EMU\_TRACE=1 -j32 # !!!NOTE MinimalConfig 只是用来测试环境，在实际使用时，使用默认配置，不加COFIG 配置 即为默认配置  # 使用 DiffTest 再次进行验证 ./build/emu -i \$AM\_HOME/apps/hello/build/hello-riscv64-xs.bin --diff ready-to-run/riscv64-nemu-interpreter-so \`\`\`  现在的香山 RTL id="uaaec23ac" class="ne-image">  这表明, DiffTest 在 PC \`0x0080000078\`处发现了测试设计 (DUT, 也就是香山处理器核) 和参考设计 (REF, 也就是 NEMU 指令集模拟器) 在 \`a0\`寄存器出现了不一样的结果 (我们认为 REF 是正确的实现, 所以 \`right\`表示 REF 给出的值, 而 \`wrong\`表示 DUT 给出的值). 因为出现了错误, 所以程序的执行被迫停止. 在香山仿真中, 仿真程序还会列举出所有寄存器的内容, 为调试错误提供帮助. 在最后, 请记得将刚才植入错误的文件恢复原样:  \`\`\`bash # 恢复 Alu.scala, 这一步将撤销我们先前手动注入的错误 git restore
